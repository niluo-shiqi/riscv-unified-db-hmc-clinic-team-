# UDB Documentation Site ‚Äî Decision Log

This file records significant decisions made during the design and implementation of the UDB
Docusaurus documentation site. Each entry captures what was decided, why, what was implemented
as a result, and what an agent would need to change to reverse the decision.

Decisions are numbered in the order they were made. Open Questions in the implementation plan
that get resolved should be recorded here.

---

## D1 ‚Äî Docusaurus root location: `doc/`

**Status**: Decided
**Resolved from**: Q1
**Date**: 2026-03

### Decision
Use `doc/` as the Docusaurus project root. All Docusaurus config files (`docusaurus.config.js`,
`sidebars.js`, `package.json`), source directories (`docs/`, `src/`, `static/`), and planning
docs (`planning/`) live under `doc/`.

### Rationale
- Keeps documentation source and site infrastructure together in one directory.
- Markdown files under `doc/docs/` are readable on GitHub as-is without running the site.
- Avoids cluttering the repo root with `node_modules/`, `package.json`, etc.
- `doc/planning/` is excluded from Docusaurus content via the `exclude` config option.

### Implemented in
- `doc/README.md` ‚Äî describes the directory structure
- `doc/planning/README.md` ‚Äî references this location
- `doc/planning/documentation-implementation-plan.md` ‚Äî all path references use `doc/` as root
- All Phase 0 tasks assume `doc/` as the Docusaurus root

### Reversal: move Docusaurus to the repo root
If this decision were reversed (Docusaurus at repo root instead of `doc/`):

1. Move all Docusaurus scaffold files from `doc/` to the repo root:
   - `doc/docusaurus.config.js` ‚Üí `docusaurus.config.js`
   - `doc/sidebars.js` ‚Üí `sidebars.js`
   - `doc/package.json` ‚Üí `package.json`
   - `doc/docs/` ‚Üí `docs/`
   - `doc/src/` ‚Üí `src/`
   - `doc/static/` ‚Üí `static/`
2. Update `.gitignore`: change `doc/node_modules/` ‚Üí `node_modules/`.
3. Update all CI workflow steps: change `cd doc && npm ...` ‚Üí `npm ...`.
4. Update `doc/README.md` to reflect the new location (or delete it).
5. Update `doc/planning/README.md` path references.
6. Update all path references in `documentation-implementation-plan.md` (search for `doc/docs/`,
   `doc/src/`, `doc/static/`, `doc/sidebars.js`, `doc/docusaurus.config.js`).
7. Decide what to do with `doc/planning/` ‚Äî move to `planning/` at repo root or keep in `doc/`.

## D2 ‚Äî TypeScript over JavaScript

**Status**: Decided
**Resolved from**: Q2
**Date**: 2026-03
**Note**: Initially decided as JavaScript; reversed to TypeScript before implementation began.

### Decision
Use TypeScript for all Docusaurus React components and custom pages. Bootstrap with
`--typescript`. All component files use `.tsx`; all non-component JS config files
(`docusaurus.config.js`, `sidebars.js`) remain plain JavaScript as Docusaurus generates them.

### Rationale
- TypeScript provides better IDE support and catches type errors in React components.
- Docusaurus has first-class TypeScript support; the `--typescript` scaffold is well-maintained.
- The additional friction for Ruby-primary contributors is low in practice ‚Äî they are unlikely
  to edit React components directly.

### Implemented in
- `doc/planning/documentation-implementation-plan.md`:
  - Task 0.1.2: `--typescript` flag in scaffold command
  - Task 0.7.3: `IDLPageHeader.tsx`
  - Phase 1 header: `src/pages/index.tsx`
  - Q2 resolution note

### Reversal: switch back to JavaScript
If this decision were reversed (back to JavaScript):

1. Re-scaffold or manually convert:
   - Rename all `doc/src/**/*.tsx` ‚Üí `.jsx` and `doc/src/**/*.ts` ‚Üí `.js`.
   - Rename `doc/src/pages/index.tsx` ‚Üí `index.jsx`.
2. Remove TypeScript dependencies from `doc/package.json`:
   - Remove `typescript`, `@docusaurus/tsconfig`, and any `@types/*` packages from `devDependencies`.
3. Delete `doc/tsconfig.json` (generated by `--typescript` scaffold).
4. In `doc/docusaurus.config.js`, remove any TypeScript-specific plugin config if present.
5. Update `documentation-implementation-plan.md`:
   - Task 0.1.2: change `--typescript` ‚Üí `--javascript`
   - Task 0.7.3: change `IDLPageHeader.tsx` ‚Üí `IDLPageHeader.jsx`
   - Phase 1 header: change `index.tsx` ‚Üí `index.jsx`
   - Q2 resolution note: update to reflect JavaScript decision

## D3 ‚Äî npm workspaces: Docusaurus as a workspace member of the root package.json

**Status**: Decided
**Date**: 2026-03

### Decision
Docusaurus dependencies are managed via npm workspaces. The root `package.json` declares
`"workspaces": ["doc"]`. There is a single `node_modules/` at the repo root and a single
`package-lock.json`. `doc/package.json` remains as the Docusaurus-specific manifest (listing
Docusaurus dependencies) but has no lockfile of its own.

All npm commands are run from the repo root:
- `npm install` ‚Äî install all dependencies
- `npm run build --workspace=doc` ‚Äî build the docs site
- `npm run start --workspace=doc` ‚Äî start the dev server

### Rationale
- The repo already had a root `package.json` (for Antora, Prettier, etc.); a second `package.json`
  in `doc/` would create two separate `node_modules/` trees and two lockfiles to maintain.
- npm workspaces unify dependency management with no loss of isolation between the root packages
  and the Docusaurus packages.

### Implemented in
- `package.json` ‚Äî added `"private": true` and `"workspaces": ["doc"]`
- `doc/package.json` ‚Äî unchanged (remains the Docusaurus manifest)
- `doc/README.md` ‚Äî updated to show workspace commands (`npm run build --workspace=doc`)
- `.gitignore` ‚Äî comment noting that `doc/package-lock.json` is intentionally absent

### Reversal: split back to a standalone doc/package.json
If this decision were reversed (Docusaurus managed independently in `doc/`):

1. Remove `"workspaces": ["doc"]` from root `package.json`.
2. Remove `"private": true` from root `package.json` if it was not there before.
3. Run `npm install` from the repo root to regenerate the root lockfile without the workspace.
4. Run `npm install` from inside `doc/` to generate `doc/package-lock.json`.
5. Update `doc/README.md`: change workspace commands back to `cd doc && npm ci` / `npm start`.
6. Update `.gitignore`: remove the comment about `doc/package-lock.json`; add `doc/node_modules/` explicitly if the bare `node_modules` entry is removed.
7. Update CI workflow to `cd doc && npm ci && npm run build` instead of `npm run build --workspace=doc`.

## D4 ‚Äî CI: Build-only, no deployment (until Phase 15)

**Status**: Decided
**Resolved from**: Q4 (partial ‚Äî resolved the "how should CI work now" part; the "how to merge with pages.yml" part is Phase 15)
**Date**: 2026-03

### Decision
The `build-docs-site` CI job builds the Docusaurus site on every PR and push to `main` as a smoke
test, but does **not deploy** it to GitHub Pages. Deployment is wired up in Phase 15 when content
is ready. The job runs inside the Docker container via `./bin/npm` (the project's wrapper that
works in both container and native mise environments).

### Rationale
- Catches build errors (broken links, TypeScript errors, missing dependencies) on every PR.
- Does not interfere with the existing `pages.yml` deployment of generated artifacts (spec docs,
  API docs, ISA Explorer, etc.).
- Docusaurus and the generated artifacts will be merged into a single GitHub Pages deployment in
  Phase 15 once content is complete; no need to complicate the deployment flow now.

### Implemented in
- `tools/test/regress-tests.yaml` ‚Äî added `build-docs-site` test with `ci_stage: pr`, `tags: [smoke]`
- `.github/workflows/regress.yml` ‚Äî auto-generated from the above
- `bin/npm` ‚Äî created as a wrapper script (matches existing `bin/node`, `bin/ruby`, etc.)
- `doc/planning/documentation-implementation-plan.md` ‚Äî task 0.4.1 marked `[x]`

### Reversal: deploy immediately (skip build-only phase)
If this decision were reversed (deploy on every push to `main` instead of waiting for Phase 15):

1. Change `ci_stage: pr` ‚Üí `ci_stage: merge_queue` in `tools/test/regress-tests.yaml` for the `build-docs-site` job.
2. Add `gh_save_artifact` to the job to save the `doc/build/` directory as an artifact (name it `docusaurus-site`).
3. Regenerate the workflow with `./bin/chore gen regress`.
4. Update `pages.yml` to download the `docusaurus-site` artifact and place it at `_site/` (see Phase 0.4.3 for the merge plan).
5. Update Q4 resolution in the implementation plan to reflect immediate deployment.

## D5 ‚Äî IDL syntax highlighting: Prism definition as canonical source

**Status**: Decided
**Resolved from**: Phase 6.5
**Date**: 2026-03

### Decision
IDL syntax highlighting is implemented as a Prism language definition (`doc/src/prism/idl.js`), which is the single canonical source. The TextMate grammar used by the VSCode extension (`tools/vscode/idl/syntaxes/idl.tmLanguage.json`) is auto-generated from the Prism definition via `bin/chore gen vscode-idl`. The Prism definition is registered with Docusaurus via a swizzled `prism-include-languages.js`.

### Rationale
- VSCode uses TextMate grammars; Docusaurus uses Prism ‚Äî these are incompatible formats with no shared source possible without a conversion step.
- Shiki (which natively consumes TextMate grammars) was considered but is not supported by Docusaurus 3.9.x as a built-in highlighter. Using `@shikijs/rehype` as a rehype plugin was investigated but would conflict with Docusaurus's built-in Prism rendering and add ~1 MB WASM to the bundle.
- The Prism definition is the natural source for the web context; the conversion to TextMate is mechanical and fully automated.
- The `bin/chore gen vscode-idl` command (and `bin/chore gen all`) regenerates the TextMate grammar automatically. The `-f` (fail-on-change) flag enables CI enforcement.

### Implemented in
- `doc/src/prism/idl.js` ‚Äî canonical Prism language definition (module.exports function form)
- `doc/src/theme/prism-include-languages.js` ‚Äî Docusaurus swizzle that registers the IDL language
- `tools/node/idl-grammar-gen/index.js` ‚Äî Node.js generator script
- `tools/vscode/idl/syntaxes/idl.tmLanguage.json` ‚Äî auto-generated TextMate grammar (do not edit by hand)
- `bin/chore` ‚Äî added `do_gen_vscode_idl` function and `vscode-idl` case; added to `do_gen_all`

### Reversal: make TextMate the canonical source
If this decision were reversed (TextMate grammar as canonical, Prism generated from it):

1. Write a TextMate ‚Üí Prism converter (reverse of `tools/node/idl-grammar-gen/index.js`).
2. Update `bin/chore gen vscode-idl` to run the new converter instead.
3. Edit `tools/vscode/idl/syntaxes/idl.tmLanguage.json` directly to change token definitions.
4. Remove the `THIS IS THE CANONICAL SOURCE` comment from `doc/src/prism/idl.js` and update it to note it is generated.

## D6 ‚Äî IDL landing page as `docs/idl/index.mdx` with card-based layout

**Status**: Decided
**Date**: 2026-03

### Decision
The IDL section has a dedicated landing page at `docs/idl/index.mdx` (sidebar_position: 0) that serves as the entry point for the entire IDL section. It uses a card-based layout with two sections: Reader Guides (audience-oriented entry points) and Language Reference (page-by-page index). The page uses `react-icons` (Phosphor icon set) for card icons and inline JSX components for the card and badge layout.

IDL has its own top-level navbar entry pointing to `/docs/idl`, separate from the Documentation dropdown.

### Reader Guides pages
- `guide-for-c-users.mdx` ‚Äî IDL for Programmers (C-family language lineage)
- `guide-for-verilog-users.mdx` ‚Äî IDL for Verilog Users
- `common-misunderstandings.mdx` ‚Äî Common mistakes
- `for-spec-writers.mdx` ‚Äî Writing new instructions and CSRs
- `quick-reference.mdx` ‚Äî Syntax cheat sheet (placed first in Language Reference, not in Reader Guides)

### Rationale
- A card-based landing page provides immediate navigation to the right entry point for each audience without requiring users to read sequentially.
- Reader Guides and Language Reference serve distinct audiences (new readers vs. existing users looking up syntax) and benefit from visual separation.
- `react-icons` (Phosphor set via `react-icons/pi`) provides consistent, semantically matched icons with no custom SVG work required.
- Quick Reference belongs in Language Reference, not Reader Guides ‚Äî it serves users who already know the language, not those learning it.

### Implemented in
- `doc/docs/idl/index.mdx` ‚Äî landing page
- `doc/docs/idl/guide-for-c-users.mdx`
- `doc/docs/idl/guide-for-verilog-users.mdx`
- `doc/docs/idl/common-misunderstandings.mdx`
- `doc/docs/idl/for-spec-writers.mdx`
- `doc/docs/idl/quick-reference.mdx`
- `doc/src/css/custom.css` ‚Äî `.idl-landing-hero`, `.idl-section-label`, `.idl-card` (hover state)
- `doc/docusaurus.config.ts` ‚Äî IDL navbar item; IDL removed from Documentation dropdown
- `doc/package.json` ‚Äî added `react-icons ^5.6.0`

### Reversal: remove dedicated IDL landing page
1. Delete `doc/docs/idl/index.mdx`.
2. Update the IDL navbar item to point to `/docs/idl/overview` instead of `/docs/idl`.
3. Remove the `.idl-landing-*` CSS classes from `doc/src/css/custom.css`.
4. Optionally remove `react-icons` from `doc/package.json` if no other pages use it.
